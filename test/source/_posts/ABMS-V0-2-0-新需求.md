本文会汇总 `ABMS V0.2` 的新的需求，主要来源于决赛赛题。

<!-- more -->

新的需求的主要技术要点是：

> 使用BasicObject、VersionObject、分类管理、版本服务、编码生成器等xDM-F内置的抽象数据模型和功能，实现miniAPP功能要求提及的版本管理、分类管理、扩展信息等功能。

要修改的实体包括设备，物料，物料分类，BOM。

## 设备扩展属性

### 设备扩展属性 问题

前期我们完成了设备的扩展属性功能，这个功能是使用的 iDME 的内置功能实现的，问题不大。

新增的功能如下：

> 预置设备类型、各类型对应的扩展信息等基础数据，以支持创建设备时的选择类型信息，创建设备、更新设备时录入相应的扩展信息。

这个功能要求创建并记录设备的模板，扩展属性的配置和各个属性的默认值。

比如 `立式加工中心` 有如下的模板

![1776766506223](/images/posts/abms-v0-2-0/1776766506223.png)

基本信息是在实体属性里面定好的，但是扩展属性需要在属性库中由用户自定义。模板中要包含扩展属性的配置信息，在这个例子中，扩展属性配置要包括工作台尺寸（字符串类型），X/Y/Z轴行程（字符串类型），主轴转速（字符串类型） 等等。

### 设备扩展属性 解决方案

比较简单的想法是创建一个 JSON 文件，如下所示：

```json
{
 // 基础属性默认值
  "no": "MC-VMC-001",
  "name": "立式加工中心",

 // 扩展属性创建 + 赋值请求
  "extAttrsRequests": [
    {
      "nameEn": "voltage",
      "name": "电压",
      "type": "DECIMAL_WITH_PRECISION",
      "description": "设备电压参数",
      "value": "4.5"
    },
    {
      "nameEn": "workspaceSize",
      "name": "工作台尺寸",
      "type": "STRING",
      "description": "工作台的桌面大小",
      "value": "850 × 500 mm"
    }
  ],

 // 其他默认值
}
```

为了方便存储，可引入 **设备模板实体 (EquipmentTemplate)** 其继承于 `Equipment` 实体，新增属性 `extAttrsRequests`，其为 JSON 类型。

这个新增的属性到后端，做的事情就是，首先调用创建扩展属性接口，将 `nameEn`，`name`，`type` 和 `description` 作为请求参数进行创建，在属性库中创建并且建立连接，然后提取基础属性和扩展属性的值，将默认值填入新增设备的对话框中，由用户自己决定修改哪些值然后用户自己提交。

为了管理设备模板实体，需要对设备模板进行 CRUD 操作的基本维护，还有应用模板，从模板创建等功能。

由于之前写好了通用 CRUD 接口，只要数据层定下来，就不需要多余的后端支持，前端可以按照上面的思路做参考做 UI 设计，但不限于这个思路。

## 物料扩展属性

### 物料扩展属性 问题

我们之前维护了 `PartCategory` 物料分类的树形结构，并将物料分类绑定到了对应物料实例中。新的需求中，不仅要让物料支持扩展属性，还要做模板化处理：

> 预置Part分类树、分类对应的扩展信息等基础数据，以支持创建Part时的选择分类信息，创建Part、更新Part、修订Part时录入相应的扩展信息。

![1776773340459](/images/posts/abms-v0-2-0/1776773340459.png)

和设备实体类似，不多做赘述。

### 物料扩展属性 解决方案

我们引入**物料模板实体 (PartTemplate)**，其继承于 `Part` 实体多了一个 `extAttrsRequests`，其为 JSON 类型。

实现思路可以参考**设备模板实体 (EquipmentTemplate)**。

在题面中，*分类*和我们的模板概念大致相同，采用我们的设计，一个*分类*可以支持多个模板，而在题面上，一个*分类*似乎只对应一个模板。我们的设计相对来说应该合理一些，并且更加灵活，还不需要修改原有的 `PartCategory`。大家可以思考一下，就 "齿轮" 这个分类，其规格有 "大齿轮"，"小齿轮"，扩展属性中，齿数这个属性就可以有两种不同的预设值/默认值。

这个设计看似多了功能会带来麻烦，实际上可以尝试使用通用化的方法做 "模板-实例"。看大家的水平了，在需求阶段已经设计好了，在编码阶段就没必要堆代码了。

1. 后端支持。由于我们之前没有做 `Part` 的扩展属性支持，所以我们要先让 `Part` 支持扩展属性。现状是，后端对实体扩展属性的配置是停留在硬编码阶段的，接下来考虑做的是，调用 iDME 系统接口，将这个配置信息按实体号拉取出来，实现通用化的扩展属性配置。在 `OpenAPI` 的接口层面上之前已经做好了扩展属性的通用化的接口设计，只需要后端继续支持到其他的实体，就可以使用了。
2. 前端支持。实际上接口已经定义好了，接口的可用性还得后端实现才能做。介于之前的经验，大家似乎不大会弄模拟接口，可以先弄个界面。建议提前弄清楚接口的返回值是什么，格式是怎么样的。

我希望大家能实现的是一个通用化管理的方法，能为智能体节约一倍的上下文。

## BOM

在题面上，一个 BOM 的结构是这样的：

![1776775561258](/images/posts/abms-v0-2-0/1776775561258.png)

我们之前的 BOM 是采用了 BOMInput 和 BOMOutput 两类的关系。我们只需要在 BOMInput 关系添加字段子项编码，位号维护即可。不需要更多的后端接口。

有关位号是什么，可以理解为电工里面的元件代号，如

![1776775758497](/images/posts/abms-v0-2-0/1776775758497.png)

这个里面，R4 和 R5 都是电阻的意思，L3 是电感，这个 R4，L3 的代码就叫位号。我们的 BOM 里面的物料是包含了完整信息的，但是在电路图的设计上会使用位号来简化。
